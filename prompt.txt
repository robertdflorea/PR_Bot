        ### 1. Overview

        The project is a **data labeling project**. Your primary role is a **PR Reviewer**. You will **not code**, but rather interact with two AI models to guide them toward providing **production-ready code** that is good, tested, and clear. In this project, your goal will be to collect human preference data on an LLM's ability to write **production-ready code**, at the level at which if the code was in a Pull Request it would be approved.

        Your task includes:

        1. **Finding or creating a suitable issue**.
        2. **Preparing the codebase.**
        3. **Interacting with the AI to solve the given issue and selecting the best answer.**
        4. **Submitting the task on the HFI Tool and on Revelo's platform.**

        ---

        ### 2. What You'll Be Doing

        In each task, you will select a codebase, that is a git repository, and ask the model to perform a single task in that codebase. Your goal is to, over multiple turns, iterate on the model's solution for that task until it reaches a production-ready state. In addition to iterating on the model's solution, you will guide the models to ensure it is working like a real engineer would. This means that you need to make sure the model is reviewing the code it wrote, validating code against task requirements, committing regularly, etc.

        ---

        ### 3. What is production-ready code?

        The question that should always be in your mind when determining if the model's output is "production-ready" is: "If the model was my co-worker and presented me with this code in a PR, would I happily approve the PR?" Until the answer to that question is yes, you should continue to iterate on the code.

        Production-ready code should:

        - Implement the feature requested thoroughly and completely
        - Consider legitimate edge cases and handle them
        - Consider security implications and handle security holes in its implementation where appropriate given the task being completed
        - Should not contain comments that add no value, are just the model's chain of thought, explain what the model just did (e.g. "# removed code because it's no longer used"), or explain exactly what the code does (instead of the why), or explain obvious things
        - Should match the style of the existing code (which is why it's extremely important for you to familiarize yourself with the codebase)
        - Should be well factored, using reasonable abstractions that match the style of the codebase (e.g. don't use classes in a purely function codebase, or vice versa)
        - Should write comprehensive tests in the style of the codebase if the codebase has tests or the task is to add tests to a codebase with no tests

        ### 1.1 Configure scrolling for the tmux terminal (click to expand)

        There is an issue you may encounter when trying to scroll the `tmux`terminal. By default, scrolling won’t work in it. However, there is a workaround if you are using macOS or Linux.

        **Edit the following file:**

        ```
        nano ~/.tmux.conf
        ```

        **Then add:**

        ```
        set -g mouse on
        set -g history-limit 75000
        ```

        **Reload the configuration:**

        ```
        tmux source-file ~/.tmux.conf
        ```

        ### 1.1.1 I got an error message running the above command (click to expand)

        If you receive a error message saying: `no server running on /tmp/tmux-XXX/default`. You may fix it by following this steps:

        ```bash
        mkdir -p /tmp/tmux-XXX
        chmod 700 /tmp/tmux-XXX 
        tmux new-session
        ```

        Re-run the `tmux source-file ~/.tmux.conf`

        ### **What are the key requirements for selecting a suitable repository?**

        The repository must be:

        - In **Python, Javascript or Typescript**.
        - Be **executable**, have correct dependencies, and ideally have tests.
        - The same repository should **not be used repeatedly**.
        - **Crucially, the repository size must not exceed 200 MB** after cloning, as too large a repository will prevent task completion by slowing the model.

        ### **What defines a good issue for this project?**

        A good issue challenges the model at a production-grade level. We look for:

        - A brand new feature, an extension of an existing feature, a refactor, a reimplementation, or the addition of tests.
        - The solution PR must have changed **at least four code files** (tests count).
        - The changes must implement something real — not just minor alterations like adding a comma or changing documentation.

